Master Thesis Log

Live control of a physically plausible virtual biped, using the Kinect

Project description

This project has three main goals:

  • Live control of a virtual biped, using a video-based motion capture solution, specifically a Microsoft Kinect sensor.
  • Controlling a physically plausible virtual character, tracked using internal plausible joint torques.
  • Robust and balanced motion, including under limited external perturbation and interaction with the virtual environment.
Kinematic challenges include missing joints, noisy data and dynamic character proportions, imposed by using a Kinect sensor. Dynamics challenges include balance, maintaining global position and orientation and preventing foot sliding. Experiments will test live control, robustness with different types of motions and both passive and active interactions with the virtual environment.

This project was preluded by a small project, experimentally assessing the quality of Kinect-generated motion data, when used to drive a physics-based character. The report of this small project (with some personal details removed) can be found here (.pdf)

17-05-2013: Goal driven IK and first 'balanced' physics character

The below video shows the initial (im)balanced motions, performed by the physically plausible character. Various changes to the implementation were made to accomplish this:

  • The IK method uses the previous goal pose as input to calculate the difference to the new goal pose, as returned by the Kinect. The previous approach used the current physics character pose as input, which resulted in the character collapsing and being unable to maintain the goal poses.
  • The physics character uses proportions similar to the Kinect data, so the absolute positions of end-effectors can be tracked as-is. The skeletal configuration of the physics character differs from the Kinect character, as some Kinect bones could be described by a single physics body volume.
  • The IK method uses the skeletal configuration of the physics character, and maps the Kinect data (shown in left two characters) to it by mapping equivalent parent-to-child bone relations.
  • The entire solution consistently describes bone orientations relative to the parent bone's coordinate system.
  • Simplified PD-controllers (without velocity tracking) track the goal pose (shown as right-most character) and customizable gains are manually set to allow the physics character to track the goal pose faithfully, using internal joint torques. The gains are roughly set to avoid extreme joint torques, which would result in twitching/oscillating motions. Implementing full PD-controllers and further tweaking of the gains is going to be required for more precise tracking control.
  • A larger than realistic ground-contact area of the feet was required to achieve the balanced physics character during an idle pose.

The video shows that the physics character can remain standing during idle motions, after a short initialization period where the character is positioned while floating slightly above the ground. The character does exhibit swaying of the arms and entire body, giving the impression of instability. When anything more challenging than idle motions are performed, the swaying worsens and the character easily loses balance.

Besides improving the implementation of the PD-controllers, another possible improvement to the model can be found in explicitly controlling the ankle orientation to move the center of mass (COM) above the ground support. Currently, the ankle orientation is set so the feet are always parallel to the ground in the goal pose. Assuming the feet are flat on the ground, the ankle orientation can be adjusted so the vector from the center of the ground support to the COM points straight up, thus making sure the COM is not off to one side of the ground support, which risks the physics character falling to that side. This balance strategy is far simpler than other approaches, like virtual forces, which means it can be tested more easily than more elaborate balance strategies. By only controlling the ankle joints however, the utility of this approach could also be limited.

09-02-2013: Keeping it Simple

As the video below demonstrates, simplifying an approach can benefit it noticeably. Compared to the IK approach of the previous log post, the approach used with the far left character is far simpler. The IK implementation still uses damped least squares IK with the Kinect SDK generated joint angles as secondary objective. However, the pelvis is no longer an end-effector, which means the remaining end-effectors are the ankles, the wrists and the base of the neck.

All joint chains use the pelvis as a base, but the position of the pelvis is not tracked directly. Instead, before and after the IK posing, the skeleton is positioned according to its support base. This means the position of the pelvis is not guaranteed to follow the Kinect data, but the secondary objectives of the IK approach (the joint angles) seem to do a very good job at keeping the pelvis position reasonable.

This much simpler approach solves most of the issues of the previous approach:

  • When the support base changes, the pelvis remains stable. Only if there is a large difference between the Kinect data of the support base and the actual support base, does a jump of the skeleton appear. This can be solved by positioning according to the actual support base, instead of the Kinect data.
  • The angles of the legs generally follow the secondary objectives, because each joint in the legs is only involved in one joint chain, simplifying the IK configuration of the legs significantly.
  • The skeleton pose is relatively stable, and little swaying can be observed. The feet are also stable on the ground, with little to no visible sliding.

Although the results of the current approach are solid, some drawbacks can still be observed:

  • The position of the pelvis is not tracked directly in this IK approach, which results in the pelvis position being influenced slightly by movement of the legs, and generally not following the Kinect data perfectly faithfully. The resulting poses are visually plausible however, and the position of the pelvis (and the entire skeleton as a result) never differs drastically from the Kinect data.
  • When both feet are on the ground, the position of the support base is the average of both ankles. This leaves some freedom for the invidivual ankles to differ from the Kinect data, as only the average of both is tracked. This means that one foot can drop through the floor slightly. This is solvable by defining the support base better.

Overall, this approach is simpler, more robust, and with minor adjustments looks to form a good basis when tracking a physics simulated character.

04-02-2013: Support Base Inverse Kinematics Configuration

The end effector configuration for the damped least squared IK has been changed, and can be seen in the image below. The colored lines denote joint chains, with the squares denoting end-effectors being tracked, and the circles denoting the base joint(s) of each joint chain. When one foot is lifted off the ground, the support base moves from an average of both ankle positions, to the ankle position of the foot still on the ground. The joint chain of the lifted ankle shortens to a base at the pelvis joint and the joint chain of the pelvis only has one base at the other ankle.

The video below shows this IK approach on the far left skeleton, with the support footing shown as pink ankle joints on the far right skeleton. Several issues resulting from using the support footing as the base for the entire skeleton can be observed:

  • When the support base changes, from two to one foot and to a lesser degree vice versa, the pelvis joint jerks significantly, resulting in the entire skeleton jerking and having to recover to a stable pose.
  • The angles of the legs frequently don't follow the Kinect data, instead showing odd knee positioning and occasionally snapping to the desired pose, causing the body to jerk. This may be caused by having up to three joint chains affecting each of the joints in the legs (the pelvis and both of the ankles).
  • In general, the skeleton's position and pose are less stable than the previous IK approach.
Experiments were performed with using the actual Jacobian nullspace for the joint angle secondary objective. But with this end-effector configuration, the nullspace was too restrictive to be able to follow the joint angles faithfully, especially in the legs. So for the video below, the pseudo-nullspace (I - jacobian^-1 * jacobian) was used instead. This means the position of the end-effectors can be (slightly) affected by the secondary objective.

Several adventages of this approach over the previous IK approach (with a pelvis base) can also be seen in the video:

  • The feet are always on the ground when they are in the Kinect data.
  • The position of the feet is much more stable relative to the ground, using this IK approach.
  • The hip bones actually rotate to fit the end-effectors to their absolute positions, which does not seem to happen in the Kinect joint angle data.
  • The joint angle rotations this IK approach calculates, actually move the body relative to the world through its support base, which is similar to the way a person uses ground reaction forces to position his/her body.

The next steps are to determine if any of the issues with this approach can be mitigated, and trying the approach with a physically simulated character.

21-01-2013: Damped Least Squares Inverse Kinematics with Weighted Joint Angle Secondary Objective

By weighting the joint angle secondary objective of the IK method (w = 0.4 in the video), a more stable result is achieved, compared to the previous results (of 17-01). However, when the Jacobian is near-singularity, pseudo inverse IK still results in instabilities in the joints. Damped least squares IK (with an appropriate damping constant: 0.3 in the video), solves these instabilities almost completely.

One issue still present is the fact that the pelvis is currently used as the base for the IK chains, but the absolute position of the pelvis can be inappropriate for the skeleton scale, resulting in a floating skeleton. So the next course of action will be to use a IK base that is more representative of the physical situation, like using support feet as a base to move the pelvis using IK.

Another issue which is likely to be causing some of the twitching movement in the below video, is the fact that the 3 DOF of the joints is separated into 3 orthogonal axes with independent rotations for the Jacobian construction in this IK implementation. The rotations are combined into a Simultaneous orthogonal rotation angle (SORA). Although this works, representing the secondary objective joint angles in the same manner introduces some limitations and instabilities.

17-01-2013: Jacobian Pseudo Inverse Inverse Kinematics and Following a Suggestion

The first inverse kinematics (IK) technique implemented is the (relatively) simple jacobian pseudo inverse. The below video shows the result on the left, when the ankles and wrists are used as end-effectors for the IK implementation, and the Kinect joint angles are used as a secondary objective. The base is the pelvis joint, whose absolute position is set directly according to the Kinect data. This is not the ideal solution, as it means the character can float, if the pelvis position is sufficiently high above the ground. This effect can be observed in the video.

The pseudo inverse IK technique does not take into account the current pose, which causes severe temporal instabilities when the limbs are stretched and the Jacobian is near-singularity as a result. The video shows these anomalies in the pose. The next IK implementation, damped least squares, does take into account the current pose, which should solve some of the worst issues with the technique shown below.

21-12-2012: Comparison of joint angles and joint positions from raw Kinect data

The video comparison below shows some of the kinematic challenges, when using a Kinect for motion tracking. When using absolute joint positions, the character remains relatively stable and its end-effectors are stable and accurate, when being correctly tracked. However, the distance between neighbouring joints varies from pose to pose. This means that joint positions, as returned by a Kinect, are not a sufficient kinematic target for a virtual character with fixed dimensions.

The Kinect for Windows SDK has functionality to convert those joint positions to both absolute and hierarchical joint angles. The character on the left in the video has fixed dimensions (bone lengths) and uses hierarchical joint angles. The global position of that character still uses absolute position data, but the joints are positioned relative to each other. As is apparent from the video, this method causes the character's end-effector positions to deviate significantly from the absolute positions seen on the right character. The left character floats or drops through the floor at certain poses.

The next course of action is to look into inverse kinematics techniques, to combine these two methods and get the best of both worlds, by tracking the absolute end-effector positions, while using the joint angles to drive the virtual character's pose towards that of the person standing in front of the Kinect.
